# Curriculum evidence matrix

This matrix is the durable course-level conformance record. A row becomes
`covered` only after its learner-facing explanation, runnable demonstration,
practice feedback, and any required independent application have been
verified. Topic names and incidental use do not count as evidence.

The final reconciliation uses `course-manifest.json`, runnable samples,
solution-selected MTP tests, starter-feedback checks, the reading-log acceptance
suite, the coverage gate, and the root Release workflow. Exact validation
results are recorded in [CONFORMANCE.md](CONFORMANCE.md).

## Cross-course area reconciliation

These area rows complement the instructional-unit evidence above. They prevent a
topic mention in one place from substituting for the connected teaching path.

| Area | Status | Target evidence and scope rationale |
| --- | --- | --- |
| Program execution, syntax, and control flow | covered | Units 01-03 name, explain, demonstrate, practice, and apply the execution and control-flow model. |
| Scalar values, nullability, conversions, and Boolean conditions | covered | Units 02-03 plus nullable analysis and downstream validation. C# requires explicit Boolean conditions rather than general truthiness. |
| Numeric representation, precision, and overflow | covered | Unit 02 connects integer division, `decimal`, binary floating point, `NaN`, and checked integral overflow to observable behavior and feedback. |
| Unicode text, bytes, raw/structured data, and I/O | covered | Units 02, 11, and 13 connect object -> JSON text -> UTF-8 bytes -> stream/file/HTTP through explanation, runnable behavior, practice, and capstone use. |
| Collections, mappings, sets, iteration, mutation, copying, and equality | covered | Units 04, 07, and 09 distinguish container choice, aliases, shallow container copies, identity, value equality, and element equality before capstone use. |
| Methods, invocation, parameter passing/binding, and returns | covered | Unit 05 explains pass-by-value, reference-type mutation versus reassignment, return values, overloads, and the later `out`/Try pattern. |
| Native object and data modeling | covered | Units 07-08 cover records, structs, classes, enums, composition, interfaces, generics, delegates, and invariant boundaries. |
| Errors, resource ownership, cleanup, and lifecycle | covered | Units 05, 11-13 connect specific exceptions, `using`/`await using`, disposal before replacement, cancellation, and response ownership. |
| Files, directories, paths, streams, buffers, HTTP, persistence, and formats | covered | Units 11 and 13 plus the capstone cover the applicable file and network surfaces. The stream buffer is explicitly a transfer detail, not a representation. |
| Dates, times, durations, time zones, timestamps, and clocks | covered | Unit 07 teaches `DateOnly`, `DateTimeOffset`, UTC, `TimeSpan`, strict ISO parsing, and clock input; Units 12-13 practice durations; the capstone applies dates/timestamps. Broader time-zone conversion is outside the local app scope. |
| Projects, namespaces, packages, build/distribution, and public API documentation | covered | Units 06, 08, and 14 cover SDK projects, `.slnx`, NuGet/locks, publish artifacts, XML documentation, and the non-package boundary. |
| Idiomatic repository structure and generated-artifact boundaries | covered | Root navigation, co-located practice roles, `.slnx`, central MSBuild/package configuration, `artifacts/`, manifest, verifier, and CI follow the declared .NET workflow. |
| CLI, environment, OS/process interoperability, streams, and exit behavior | covered | Setup, Unit 14, and the capstone cover arguments, environment configuration, stdout/stderr, cancellation, and exit codes across claimed OSes. Spawning child processes is outside the course's application goals. |
| Static typing, analyzers, nullability, and native abstractions | covered | Shared build configuration and Units 02, 06-10 make compiler, nullable, analyzer, generic, and delegate feedback observable. |
| Testing, debugging, formatting, analysis, coverage, and CI | covered | Units 05, 10, and 14 plus the root workflow teach the widening lifecycle with actionable failure paths and local/CI parity. |
| Async, cancellation, concurrency, and owned lifecycle | covered | Unit 12 provides explanation, sample, practice, and feedback; Units 13-14 and the capstone apply owned async I/O and cancellation. |
| Persistence/API validation and untrusted input | covered | Units 11 and 13 plus capstone normal/boundary/failure tests validate paths, stored data, DTOs, statuses, base URLs, and strict dates. |
| Beginner-appropriate algorithms and performance | covered | Units 04, 09, and 12 teach container selection, repeated work, materialization, and bounded concurrency. Formal algorithm analysis is an explicit non-goal. |
| Theory-to-sample-to-practice-to-capstone continuity | covered | The manifest, 14 sequenced guides, solution/starter feedback, and six capstone milestones establish the complete evidence progression. |
